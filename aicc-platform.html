For exactly-once semantics we actually had to implement idempotency keys on the consumer side since RabbitMQ itself only guarantees at-least-once. like the pattern was — producer sends message, consumer checks the key in Redis, processes if unseen, marks as seen atomically.
AI 12:34
Good — idempotency keys are the right approach. I noticed in your task-queue-service GitHub repo that you're using a UUID v4 as the idempotency key. How do you handle the storage cleanup of these keys? What's your TTL strategy?
You 12:36
Right, so um we store them in Redis with a 24-hour TTL. The assumption is that if a message isn't redelivered within 24 hours, we can safely expire the key. We also run a nightly cleanup job for any keys that slip through, basically as a safety net. The bigger challenge was actually the Redis cluster failover — like if the primary goes down mid-write you could lose the idempotency record entirely...
AI 12:38
That's a real edge case. How did you solve the Redis failover problem?
You 12:39
We ended up using Redis Sentinel with synchronous replication for the idempotency keys specifically. Performance hit was acceptable since these writes were off the critical path.
